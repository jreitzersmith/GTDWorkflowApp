GTD Workflow App

Project summary · April 2026 · src/App.jsx

Overview

This project is a personal Getting Things Done (GTD) task manager built as a React single-page application. It is implemented as a single component file — src/App.jsx — using React hooks, inline styles, and Vite as the build tool. All state is persisted to localStorage, making it fully client-side with no backend required.

The application has two primary areas: a task panel on the left that displays one of seven GTD buckets (Inbox, Next Actions, Projects, Waiting For, Someday/Maybe, Completed, Inbox History), and an AI Coach panel on the right that connects to either the Anthropic Claude API or a locally-hosted Open WebUI instance. The coach operates in five modes: Chat, Process, Weekly Review, Brain Dump, and Project Review.

The entire color palette is defined in a single COLORS constant at the top of the file — each bucket has a dedicated foreground and background color. The seven buckets and five coach modes are declared as BUCKETS / COACH_MODES objects, which drive both the sidebar rendering and the system prompts.

Foundation — React App Scaffold

058bedd 81d5059 Commits: Initial commit + Vite scaffold

The project was bootstrapped with Vite 5 and the React plugin. The entry point is src/main.jsx, which mounts the single GTDManager component into #root. An index.html and vite.config.js provide the HTML shell and Vite configuration. A launch.bat file was added for double-click startup on Windows, combining npm run dev with automatic browser opening.

The task data model is established at every creation site in the code — a plain object with eight fields. Every task always carries the full shape regardless of which bucket it lives in; project-specific fields (childIds, parentId) are added only where relevant. See task data model.

Smart Inbox Processing

8d73f4c Commit: Smart inbox processing

Inbox processing is the heart of GTD: every captured item must be clarified and moved to an appropriate bucket. The AI Coach assists this with a structured process mode whose system prompt — SYSTEM_PROMPTS.process — instructs the model to classify each item, rewrite it as a concrete verb-first action, and end its reply with an exactly-formatted →ACTION: tag on its own line.

The →ACTION protocol supports five outcomes: next, project, someday, waiting, and delete. For a project outcome the tag carries three pipe-separated fields: →ACTION:project|<Project name>|<First next action>, allowing both the parent project and its initial subtask to be created in one confirm click.

Once the model replies, the parsed action is held in pendingAction state and displayed in a PendingActionBar — a confirmation strip at the bottom of the coach panel showing the reworded title before the user commits. Clicking confirm fires handleConfirmMove, which archives the original inbox item to the Inbox History bucket (rather than deleting it), creates the new task(s), and automatically feeds the next inbox item to the AI — a tight loop that processes the entire inbox without manual intervention.

Multi-Provider AI Support

28e0c25 Commit: Open WebUI provider support

Rather than hardcoding a single AI endpoint, the application supports two providers selectable from a ProviderSelector dropdown in the coach panel header: Claude (Anthropic's API direct from the browser) and Local (any Open WebUI instance on the local network). Both provider and localModel are persisted to localStorage so the choice survives page refresh.

The central callAI function branches on the selected provider. For Claude it calls https://api.anthropic.com/v1/messages with the anthropic-dangerous-direct-browser-access header (required for browser-origin requests). For Open WebUI it posts to /api/chat/completions using the OpenAI-compatible format — the same conversation history shape works for both. When the provider is switched to Local, a fetchModels() call hits /api/models to populate a live dropdown of available model IDs.

The system prompt for every call is composed at call-time by concatenating the mode-specific prompt with a formatted snapshot of the user's entire task list via getTaskContext(), so the AI always has full visibility of the current state.

Task Metadata Panel (Phase 1)

5cde89e Commit: Phase 1 — task metadata panel

Every task row in the application can expand to reveal a collapsible metadata panel triggered by a chevron button (› / ▾) on the right side of the row. The panel is rendered by the TaskRow component using local expanded state — it appears and disappears without any global state involvement.

The metadata panel — see expanded panel JSX — exposes four editable fields:

  • Location — multi-select chip buttons drawn from the global locations list. Active selections are highlighted in project-blue. Each click calls toggleLocation(), which computes the next array and calls onUpdateTask.
  • Due Date — an HTML5 <input type="date"> with dark colorScheme. A clear button (✕) sets the field back to null.
  • Priority — multi-select from four fixed values: Imperative, As Possible, Financial, External. Uses the same toggle pattern as Location.
  • Effort — single-select chip buttons (only one effort level active at a time). Active selection highlighted in teal (COLORS.effort).

When the panel is collapsed, any set values are shown as compact summary chips below the task text (or inline with the row, depending on the Tag Display setting). The chevron is always visible when any metadata is set, ensuring users can always find their way back in.

Location Settings Manager (Phase 2)

6c37280 Commit: Phase 2 — location settings manager

Locations are a user-configurable list stored in localStorage under gtd_locations, initialized to four defaults: Home, Work, Phone, Computer. A Settings button at the bottom of the sidebar swaps the task area for the SettingsPanel, which hosts the LocationManager component.

LocationManager provides a full CRUD interface for the location list. Each row shows the location name alongside a live count of how many tasks use it. Renaming is done inline — clicking "✎ Rename" replaces the label with an input; pressing Enter or clicking Save calls renameLocation(), which cascades the change across every task that carries that location. Removing a location that is in use triggers a confirmation sub-row with a replacement picker — the user can choose another location to substitute, or leave it empty to simply remove the tag. The removeLocation callback applies the substitution across all affected tasks atomically.

New locations are added via a text input at the bottom of the list. Duplicate entries are silently ignored by the addLocation callback.

Next Actions Waterfall & Group-By (Phase 3)

68ef357 053cffd Commits: Phase 3 · Group-by Project

The Next Actions bucket has two distinct filtering and display systems layered on top of each other.

Waterfall Filter

In GTD, only the next action in a sequence should be visible — not all downstream steps at once. The waterfallFilter pure function implements this rule in three lines. It always passes standalone actions (no parentId) and direct children of project tasks (level 2). Children of other next actions (level 3+) are hidden until their direct parent is marked done. This means a multi-step workflow surfaces one action at a time rather than overwhelming the list.

Group-By Toolbar

A toolbar in the Next Actions header offers six grouping modes: None, Project, Location, Due Date, Priority, and Effort. The groupByField pure function handles all modes. Multi-value fields (Location, Priority) use the first value as the group key. The Project mode is more complex — it walks the parentId chain of each task all the way to the root project, then groups by that root's text. This means even deeply-nested subtasks correctly appear under their top-level project heading.

The Effort group-by mode sorts groups by duration rather than alphabetically — effortToMinutes(a) - effortToMinutes(b) — so groups always appear shortest-to-longest (2 min → 5 min → 1 hour → 1 day). Tasks with no value for the chosen field collect at the bottom under a contextual label ("No Project", "No Effort", or "Ungrouped").

Each group is rendered by a GroupDivider header that shows the group name, a task count badge, and — when any task in the group has effort set — a teal ⏱ total badge showing the cumulative effort of the group.

Project Task Creation (Phase 4)

ffc7764 Commit: Phase 4 — project task creation

The add-task row in the Projects view is context-aware. A <select> dropdown controls which mode it operates in. When set to + New project (the default), the input creates a root project task with an empty childIds: [] array. When an existing project is selected from the dropdown, the input creates a next-action subtask that is simultaneously added to the selected project's childIds array and given a parentId pointing to that project.

The addProjectTask function handles both paths. The button label and input placeholder both adapt to the current selection — the button reads "Add Project" in the default state and "Add Task" (highlighted in project-blue) when a parent is selected. The dropdown resets to the default when the user navigates away from the Projects bucket. Completed projects are excluded from the dropdown.

Drag-and-Drop Project Tree (Phase 5)

186bf3e Commit: Phase 5 — HTML5 drag-and-drop

The Projects view renders tasks as a recursive tree using the ProjectTree component rather than a flat list. ProjectTree calls itself recursively up to 5 levels deep. At each level it calls getOrderedChildren, which for root-level tasks uses array position as the ordering signal, and for child tasks uses the parent's childIds array.

Every task row is wrapped in a draggable div with HTML5 drag event handlers wired to four callbacks in GTDManager: onDragStart, onDragOver, onDragEnd, and onDrop. The drag callbacks track two pieces of state: dragId (the task being dragged) and dropTarget (the task being hovered over and the computed drop position).

Drop position is calculated from the mouse's Y coordinate relative to the hovered element: the top third is "before", the middle third is "inside" (reparent as first child), and the bottom third is "after". A DropLine component — a 2px blue horizontal rule — renders at the correct tree depth before or after the target row to give visual feedback. The "inside" position shows a blue outline on the target row instead.

All tree mutation logic lives in the pure moveTaskInTree function. It handles every move case: reordering within the same parent, reparenting to a different project, promoting to root, and demoting under a project. A recursive isAncestorOf() guard inside moveTaskInTree prevents circular references — you cannot drop a task inside one of its own descendants.

AI Project Review (Phase 6)

879e53a Commit: Phase 6 — AI Project Review

The AI Coach can review every active project in sequence, identify missing next actions, and add them directly to the task list. The flow begins with the "Review Projects" button in the Projects panel header, which calls startProjectReview. This filters for all non-done root projects, switches the coach panel to projectReview mode, and immediately kicks off the first review.

For each project, reviewProject constructs a structured prompt — project name, metadata, and current subtask list — and sends it to the AI using the projectReview system prompt. That prompt instructs the model to end every response with a →SUGGESTIONS: block listing up to four concrete missing actions. The extractSuggestions function parses that block using a regular expression and splits it into an array of strings.

The coach panel renders a ProjectReviewBar component showing each suggestion as a pre-checked checkbox. The user can uncheck any suggestions they want to dismiss. A single adaptive button then reads "Add N & Next →" (if there are projects remaining), "Skip →" (if all are unchecked), or "Finish Review ✓" (at the last project). Clicking it triggers advanceProjectReview, which creates new subtask objects for every checked suggestion, appends their IDs to the project's childIds, and either moves to the next project or exits review mode.

Effort Metadata System

61b3b06 Commit: Effort metadata field

Effort is a single-select metadata field on every task — unlike Location and Priority which are multi-value arrays, effort is a string | null. The available effort levels are stored in localStorage as gtd_efforts and initialized to an eleven-item default list ranging from "2 min" to "1 month".

Two module-level pure functions handle effort arithmetic. effortToMinutes converts a human-readable string to minutes using substring matching ("hour", "day", "week", etc.), so user-added values like "4 hours" or "2.5 days" parse correctly without a lookup table. minutesToEffortLabel converts back to a compact label (150 → "2.5h", 1440 → "3d"). Both functions are used in two places: the group-by effort total in GroupDivider, and the projectEffortTotal computed value inside TaskRow.

The projectEffortTotal is calculated when a task is in the project bucket and is currently being viewed in the Projects panel. It maps the task's childIds to their task objects, sums their effort values via effortToMinutes, and converts the total back to a compact label. This total appears as a teal badge inline with the project name row.

The Effort Levels section of Settings is managed by the EffortManager component, which follows the same pattern as LocationManager: inline rename (Enter/Escape, cascades to all tasks), remove with a confirm dialog (clears effort from affected tasks), and an add form at the bottom.

Next Actions Enhancements

f98d445 4de6b06 Commits: Group effort totals · Effort group-by · Assign to project

Cumulative Effort in Group Headers

When Next Actions are grouped by any field, the GroupDivider for each group shows a cumulative ⏱ badge. The total is computed inline in the render loop — items.reduce((sum, t) => sum + effortToMinutes(t.effort), 0) — and only displayed when the sum is non-zero. This makes it immediately visible how much total effort is committed to a given location, project, or priority category.

Assign Unaffiliated Tasks to a Project

A 📁 button appears on hover for any Next Action that has no parentId. Clicking it reveals an inline assign panel below the task row with a dropdown listing all active root projects plus a "New project…" option. Selecting an existing project and clicking Assign fires the assignToProject callback, which sets parentId on the task and appends its ID to the project's childIds. Selecting "New project…" shows a text input; on confirm, a new project task is created with the typed name and the current task is immediately linked to it as its first child.

Inline Title Editing

988d26a Commit: Inline title editing

The first field in every task's expanded metadata panel is a Title input — a full-width text field pre-filled with task.text. The title edit section is managed by two pieces of local state in TaskRow: editTitle (the draft string) and a useEffect that syncs it from task.text whenever the task is updated externally.

The input's border turns blue as soon as the draft diverges from the saved text, acting as a dirty indicator. Save and ✕ revert buttons appear only when there is an unsaved change. Enter saves, Escape reverts. Saving an empty or whitespace-only value reverts silently rather than blanking the title. The saveTitle function calls onUpdateTask(task.id, { text: trimmed }), which propagates through the standard update path.

This feature works identically in all views — Inbox, Next Actions, Projects, Waiting For, Someday, and Completed — because TaskRow is the shared renderer across every bucket.

Tag Display Mode

6f2abf9 Commit: Inline tag display mode

The placement of metadata chips on collapsed task rows is controlled by a tagDisplay state value ("below" or "inline"), persisted to localStorage as gtd_tag_display. The setting is changed via the TagDisplaySetting component at the top of the Settings panel — two large toggle buttons, one per mode, with a short description of each.

In Below text mode (the default), chips are rendered inside the text column div under the task name, adding vertical height to rows that have metadata. In Inline mode, the chips move to a separate flex div inserted between the text column and the action buttons column in the main row, using whiteSpace: nowrap and alignSelf: center to sit on the same baseline as the task text. The two render paths are controlled by a single conditional: tagDisplay check in TaskRow. In both modes, chips disappear from the row when the metadata panel is expanded, since the full editable fields are already visible.

// Level-2 tasks (direct children of a project) always show.
// Level 3+ tasks only show when their direct parent is done.
function waterfallFilter(nextTasks, allTasks) {
  return nextTasks.filter(task => {
    if (!task.parentId) return true;
    const parent = allTasks.find(t => t.id === task.parentId);
    if (!parent) return true;
    if (parent.bucket === "project") return true; // level 2 — always visible
    return !!parent.done;                              // level 3+ — visible only when parent done
  });
}

groupByField — multi-mode grouping with effort sort src/App.jsx · lines 129–168 
function groupByField(taskList, field, allTasks = []) {
  const ungroupedLabel = field === "project" ? "No Project"
                       : field === "effort"  ? "No Effort"
                       : "Ungrouped";
  const groups = {}, ungrouped = [];
  taskList.forEach(task => {
    let keys = [];
    if      (field === "location") keys = task.location || [];
    else if (field === "priority") keys = task.priority || [];
    else if (field === "dueDate")  keys = task.dueDate ? [task.dueDate] : [];
    else if (field === "effort")   keys = task.effort  ? [task.effort]  : [];
    else if (field === "project") {
      // Walk up parentId chain to find the root project name
      if (task.parentId) {
        let cur = task;
        while (cur.parentId) {
          const parent = allTasks.find(t => t.id === cur.parentId);
          if (!parent) break;
          cur = parent;
        }
        if (cur.id !== task.id) keys = [cur.text];
      }
    }
    if (!keys.length) { ungrouped.push(task); return; }
    const key = keys[0];
    if (!groups[key]) groups[key] = [];
    groups[key].push(task);
  });
  // Effort: sort by duration (shortest first). Others: alphabetical.
  const sorted = Object.entries(groups)
    .sort(([a], [b]) => field === "effort"
      ? effortToMinutes(a) - effortToMinutes(b)
      : a.localeCompare(b))
    .map(([key, items]) => ({ key, label: key, items }));
  if (ungrouped.length) sorted.push({ key: "__ungrouped__", label: ungroupedLabel, items: ungrouped });
  return sorted;
}

effortToMinutes + minutesToEffortLabel src/App.jsx · lines 172–193 
function effortToMinutes(str) {
  if (!str) return 0;
  const s = str.toLowerCase().trim();
  const num = parseFloat(s);
  if (isNaN(num) || num <= 0) return 0;
  if (s.includes("month")) return Math.round(num * 9600); // ~4 w × 5 d × 8 h
  if (s.includes("week"))  return Math.round(num * 2400); // 5 d × 8 h
  if (s.includes("day"))   return Math.round(num * 480);  // 8 h
  if (s.includes("hour"))  return Math.round(num * 60);
  if (s.includes("min"))   return Math.round(num);
  return 0;
}

function minutesToEffortLabel(minutes) {
  if (!minutes || minutes <= 0) return null;
  if (minutes < 60)   return `${minutes}m`;
  if (minutes < 480)  return `${+((minutes / 60).toFixed(1))}h`;
  if (minutes < 2400) return `${+((minutes / 480).toFixed(1))}d`;
  if (minutes < 9600) return `${+((minutes / 2400).toFixed(1))}w`;
  return `${+((minutes / 9600).toFixed(1))}mo`;
}

function extractSuggestions(text) {
  const m = text.match(/→SUGGESTIONS:\n([\s\S]*?)$/);
  if (!m) return [];
  return m[1]
    .split('\n')
    .map(l => l.replace(/^\d+\.\s*/, '').trim())

const reviewProject = useCallback(async (project, idx, total) => {
  const children = getOrderedChildren(project.id, tasks);

getOrderedChildren + moveTaskInTree src/App.jsx · lines 209–278 
function getOrderedChildren(parentId, allTasks) {
  if (parentId === null) {
    // Root projects: ordered by position in the flat tasks array
    return allTasks.filter(t => t.bucket === "project" && !t.parentId);
  }
  const parent = allTasks.find(t => t.id === parentId);
  return (parent?.childIds || []).map(id => allTasks.find(t => t.id === id)).filter(Boolean);
}

function moveTaskInTree(allTasks, dragId, targetId, position) {
  if (dragId === targetId) return allTasks;
  // Guard: prevent dropping inside own descendant
  function isAncestorOf(ancestorId, nodeId, seen = new Set()) {
    const node = allTasks.find(t => t.id === nodeId);
    if (node?.parentId === ancestorId) return true;
    return node?.parentId ? isAncestorOf(ancestorId, node.parentId, seen) : false;
  }
  if (isAncestorOf(dragId, targetId)) return allTasks;

function ProjectTree({ parentId, depth, allTasks, dragId, dropTarget,
                         onDragStart, onDragOver, onDragEnd, onDrop, rowProps }) {
  if (depth > 5) return null; // max depth guard
  const children = getOrderedChildren(parentId, allTasks);
  if (!children.length) return null;

  return <>
    {children.map(task => {
      const isTarget   = dropTarget?.id === task.id;
      const isDragging = dragId === task.id;
      return (
        <div key={task.id}>
          {isTarget && dropTarget.position === "before" && <DropLine depth={depth} />}
          <div
            draggable
            onDragStart={e => { e.stopPropagation(); onDragStart(task.id); }}
            onDragOver={e => { e.preventDefault(); e.stopPropagation(); onDragOver(e, task.id); }}
            onDrop={e => { e.preventDefault(); e.stopPropagation(); onDrop(task.id); }}
            style={{ opacity: isDragging ? 0.35 : 1,
                     outline: isTarget && dropTarget.position === "inside"
                              ? `2px solid ${COLORS.project}66` : "none" }}
          >
            <TaskRow task={task} isSubtask={depth > 0}
                     indentOverride={depth * 22} {...rowProps} />
          </div>
          {isTarget && dropTarget.position === "after" && <DropLine depth={depth} />}
          <ProjectTree parentId={task.id} depth={depth + 1} ... />
        </div>
      );
    })}
  </>;
}
