GTD Coach Modes — Implementation Reference

Per-mode description of system prompts, context, action types, and UI flow · GTDWorkflowApp

💬 Chat General Assistant

The default mode. Open-ended GTD coaching with lazy task context, live tool use, and full action coverage.

Context Sent to API

Chat is the only mode that uses lazy task context. Instead of sending the full task list on every message, the system prompt includes a compact bucket-count summary:

[Task Overview — Today: 2026-05-21 | Inbox: 4 · Next Actions: 12 · Projects: 8 · Waiting For: 3]
Call get_task_context to retrieve full task details when needed.

The AI calls the get_task_context tool on demand when it needs specific task IDs or content. This keeps per-message token cost low. All other modes send the full task list unconditionally.

No calibration context is injected (effort calibration is reserved for process and projectMetadata modes).

Tools Available (Claude provider only)

get_task_context

Fetches the full task list filtered by optional bucket array. Called automatically when the AI needs task details.

web_search / web_fetch

Tavily-powered search and fetch. Available when VITE_TAVILY_API_KEY is set.

Gmail tools

Search, label, bulk-action, compose, send, filter management. Available when Google token is connected; write tools gated on modify/compose/send scope.

Drive tools

Search and read Drive files. Available when Drive is enabled.

get_weather

Current conditions + 24-hour forecast. Available when VITE_OPENWEATHERMAP_API_KEY is set.

Action Types

Chat mode auto-processes multiple action lines per response via the taskActionLines loop. Actions are applied immediately with no pending-action bar (no user confirmation step).

→ACTION:update

Patch any field on an existing task by ID. Supports notes_append to add to existing notes rather than replace.

→ACTION:add

Add a child task under an existing project. Parent resolved by fuzzy lookup: exact ID → exact title → substring/contains match. Supports full metadata: due, defer, effort, location, priority, category, recurrence, notes, isSomeday, isWaitingFor.

→ACTION:create

Create a standalone task with no known parent. Supports bucket:inbox|next|project. Next-action tasks go under Uncategorized.

→ACTION:link_email

Attach a Gmail message ID to a task as a Drive-attachment-style reference.

→ACTION:calendar_create/update/delete

Create, reschedule, or delete Google Calendar events. Calendar actions and task actions are mutually exclusive in one reply.

→ACTION:create-doc / create-sheet / create-slides

Generate a Google Doc (from response text), Sheets export (from live task list with filter params), or PowerPoint upload.

Response Constraints

  • Under 100 words per response before any action tags.
  • Multiple ACTION lines allowed per response — placed at the end, parent-before-child.
  • For hierarchical task creation, the AI writes a flat plain-English plan first, then emits one ACTION line per task.
  • Task and calendar actions are mutually exclusive in a single reply.
  • Gmail bulk workflow is two-phase: discovery (sample + propose query) → explicit user confirmation → execution. This prevents over-labelling.

No Pending Action Bar

Note: Chat mode does not use the pending-action confirmation bar. All task actions are applied immediately when the AI response is received. The user can undo via the task list directly.

📥 Process Inbox GTD Capture → Clarify → Organise

Processes inbox items one at a time. The AI makes a routing decision immediately; the user reviews and confirms via a metadata bar before the action is committed.

Entry Points

Full inbox processing

Clicking "Process Inbox" in the coach mode selector calls startProcessInbox(). Resets skipped-IDs set and session-created-tasks list, switches to process mode, fires the first inbox item immediately.

Single-task mode

Clicking "Ask AI" on an individual inbox task calls askAIAboutTask(). Sets singleTaskMode = true. Processes just that one task, then stops rather than auto-advancing.

Context Sent to API

Buckets: inbox, project, waiting. Projects and waiting-for items are included so the AI can detect duplicates and look up real parent IDs for →ACTION:add.

Calibration context: injected — shows configured effort labels and any per-label overrides so the AI suggests effort values using the correct label set.

User message format: Process this GTD inbox item: "task text"

History: reset to [] for each new item — every item is an independent conversation.

AI Decision Flow

1
Actionability check

Is the item actionable? If not, emit →ACTION:delete. Also delete if the item already exists in another bucket — the AI notes where it is already tracked.

2
Reword as concrete action

Rewrite the title starting with a strong verb (Call, Draft, Research, Buy, Schedule, etc.).

3
Route immediately

File to one of five destinations (see Action Types below). Routing must be immediate — the AI makes a best-guess and flags the assumption in prose. It does not ask a clarifying question unless the item is completely uninterpretable (e.g. a single ambiguous word with no context).

3a
Emit exactly one →ACTION line

Includes all inferred metadata: due date, defer date, effort, location, category, recurrence, notes. Under 80 words before the tag. The action line never appears mid-response.

Action Types (Process Mode)

→ACTION:add

Child of an existing project. Parent resolved by fuzzy lookup: exact ID match → exact title match → substring/contains match on active project titles. Inherits sibling category if consistent.

→ACTION:next

Standalone next action. Goes under Uncategorized project. Same syntax with optional someday:true or waitingFor:true flags.

→ACTION:project

New project + first next action. Optionally parented under a category or subcategory node. Accepts optional someday:true or waitingFor:true to create the project directly in Someday/Maybe or Waiting For.

→ACTION:delete

Item is not actionable, or is already tracked elsewhere. The original inbox item is moved to inboxHistory.

// Full syntax reference
→ACTION:add|<title>|parent:<id>[|due:YYYY-MM-DD][|defer:YYYY-MM-DD][|recur:FREQ:N[:DAYS]][|effort:<label>][|notes:<text>]
→ACTION:next|<title>[|due:...][|someday:true][|waitingFor:true][|effort:...][|category:...][|notes:...]
→ACTION:project|<name>|<first next action>[|parent:<category_id>][|effort:...][|someday:true][|waitingFor:true][|notes:...]
→ACTION:delete

Question Guard

After the API returns, the code checks the last non-empty line before the →ACTION: marker for a ? character (stripping markdown link brackets first). If that last line contains a question, setPendingAction is skipped — the message renders as a chat bubble and the user answers before any action bar appears.

This prevents the confirmation bar from showing up when the AI is asking for clarification rather than proposing a routing decision.

Pending Action Bar (User Review Step)

When setPendingAction fires, PendingActionBar renders at the top of the chat panel showing the proposed filing with all inferred metadata. For add and project types, the parent name is resolved from the task list and stored on the action object for display. The user has three choices:

  • Confirm (OK) — calls handleConfirmMove(). Commits the action.
  • Skip — calls handleSkipPendingAction(). Item stays in inbox, its ID added to skippedInSessionIds. Auto-advances.
  • Delete — calls handleDeleteInboxItem(). Archives to inboxHistory. Auto-advances.

The bar also supports inline metadata editing via onUpdatePendingAction — the user can change due date, effort, location, etc. before confirming without going back to chat.

Confirm Flow (handleConfirmMove)

1
Find current inbox task

Looks up the task being processed via processingTaskId.current. If null (action came from chat/review, not inbox), skips the archive step.

2
Archive original

Moves the source inbox item to bucket: 'inboxHistory' (soft archive, never deleted).

3
Create new task(s)

next/someday/waiting → single task, bucket:'project' + appropriate flag, parented under Uncategorized.
project → project node + first next-action child; optional category parent; isSomeday/isWaitingFor flags applied when present.
add → child task under specified parent; falls back to Uncategorized if parent not found.
delete → no new task.

4
Auto-advance

Clears pendingAction, then after a 300 ms delay calls processNextInboxItem on the next non-skipped inbox item. If none remain, triggers end-of-session logic.

Session End

When the inbox is cleared, sessionCreatedTasksRef is checked. If 2 or more "loose" tasks were created (next actions, someday, waiting — but not tasks placed under an explicit project via add or project), the project group suggestion bar appears in the chat panel, offering to group the new tasks under an existing or new project. Then "🎉 All items processed." is shown.

🧠 Brain Dump Capture Everything

Surfaces open loops by cycling through life areas one at a time. Each captured item is immediately added to the inbox as a task.

Context Sent to API

Full task list (all buckets, no filter). No calibration context. The task list is present so the AI can avoid suggesting duplicates, though it does not actively de-duplicate — it relies on the user's judgment.

How It Works

The AI cycles through ten life areas in fixed order, asking about each one and waiting for the user to finish before moving on:

  1. Work tasks
  2. Emails to send
  3. People to follow up with
  4. Projects falling behind
  5. Personal errands
  6. Home tasks
  7. Health commitments
  8. Finances
  9. Learning goals
  10. Anything nagging you

Action Type

A single action type is used in this mode:

→ACTION:create|<exact item text>|bucket:inbox

One →ACTION:create line per captured item, emitted immediately in the same response as the acknowledgement. The item is created directly in the inbox — no pending-action bar or user confirmation step. Multiple items mentioned in a single message each get their own action line.

Response Constraints

  • Under 60 words per response, before the action tags.
  • After each area, immediately ask about the next.
  • After all ten areas, give a closing summary listing exactly the items captured this session with a count. The count is derived from →ACTION:create lines emitted in the conversation — not from the task list context (which includes pre-existing tasks).

No Pending Action Bar

Note: Brain Dump actions are applied immediately via the taskActionLines loop (same path as Chat mode). Items go straight to the inbox. The user reviews and processes them later using Process Inbox mode.

🔍 Project Review Two Sub-Modes

Reviews every active project in sequence. After selecting a sub-mode, the AI works through the project queue one project at a time.

Entry Point & Mode Selection

Clicking Project Review in the mode selector calls startProjectReview(), which builds the review queue (all active, non-done project-bucket tasks), resets all review state, and renders a ReviewModeBar offering two choices:

Mode A — Missing Tasks

Reviews each project for missing next actions. The AI suggests 2–4 new concrete actions not already present in the subtask list.

Mode B — Metadata Quality

Reviews each project node and its active subtasks for incomplete or incorrect metadata fields (effort, due date, defer, priority, location, category, nodeType).

Review Queue

buildReviewQueue(tasks) returns the ordered list of reviewable projects. Tasks tagged type:category or type:subcategory are organisational containers and are excluded — only type:project, type:subproject, and untagged project-bucket tasks are in the queue.

Mode A — Missing Tasks

Prompt sent to API

Project N of M: "full/path/name"
Metadata: effort:2h | Due: 2026-06-01 | ...
Current subtasks:
- Subtask one [id:abc]
- Subtask two [id:def]

System prompt: projectReview. The AI writes 2–3 sentences assessing the project's state, then proposes up to 4 new actions starting with strong verbs. It must not suggest anything already present in the subtask list, even with different wording.

Response block:

→SUGGESTIONS:
1. Draft initial outline for Q3 report
2. Schedule stakeholder review meeting
3. Research vendor pricing options

ProjectReviewBar (UI)

After the API returns, extractSuggestions(reply) parses the →SUGGESTIONS: block and filters out any suggestions whose text (case-insensitive) already appears in existing subtasks. Each remaining suggestion is shown as a checkbox item, all pre-checked. The user can uncheck any they don't want.

  • Accept (Next →) — advanceProjectReview(): creates all checked suggestions as new next-action subtasks under the current project, marks the project reviewed: true, then calls reviewProject on the next queue item.
  • Skip — skipProjectReview(): marks the project reviewed, advances without adding tasks.

When the queue is exhausted, the mode reverts to chat and shows "🎉 All N projects reviewed!"

Mode B — Metadata Quality

Prompt sent to API

Project N of M: "full/path/name"
Today: 2026-05-21
Available locations: Home, Office, Phone
Available efforts: 15m, 30m, 1h, 2h, 1d
Nodes to review:
- Project Name [id:xyz] (effort:none, due:none, ...) ← container node
- Subtask one [id:abc] (effort:1h, due:none, priority:none, ...)

System prompt: projectMetadata. The AI reviews effort, due date, dueTime, defer date, priority, location, category, and nodeType for each node. It only uses values from the available lists, only suggests due dates when context strongly implies a deadline, and only suggests priority when clearly warranted.

Response block:

→METADATA:
abc123|effort:30m|category:Admin
def456|due:2026-06-01|priority:p1

MetadataReviewBar (UI)

extractMetadata(reply) parses the →METADATA: block into { taskId, fields } entries. Each entry is shown with the task name, the suggested fields, and editable text inputs for overriding values before accepting. Priority and location fields are normalised from comma strings back to arrays when applied.

  • Accept (Next →) — advanceMetadataReview(): calls updateTask(id, overrides) for each accepted suggestion, marks the project reviewed, advances.
  • Skip — skipMetadataReview(): marks reviewed, advances without applying changes.

Calibration Context

The projectMetadata prompt receives a calibration context block (same as process mode) so the AI knows which effort labels are valid and won't invent new ones.

🌅 Daily Review Start of Day / End of Day

A single button that toggles between Start of Day and End of Day modes. Phase state persists across app restarts via localStorage; it auto-resets each calendar day.

Phase Persistence

Phase is stored as gtd-daily-phase in localStorage as { phase: 'start'|'end', date: '...' }. On load, if the stored date is today, the phase is restored; otherwise it resets to 'start'. This means on the first trigger of each day the user sees Start of Day; after completing it, the button shows End of Day until midnight.

Start of Day (SoD)

Phase: start → flips to end after trigger

Pre-computed summary block

Before calling the API, the app computes counts from the live task list and injects them into the user message as a [SoD Summary] block:

[SoD Summary]
- Overdue: 3
- Due today: 2
- Due this week (>1 hr effort): 5
- Have due date but no calendar event: 1  // only if Calendar enabled
- Unprocessed inbox items: 4
[/SoD Summary]

If there are overdue items, an urgency note is appended to the user message: "You have 3 overdue items that need attention."

AI Conversation Flow (4 steps)

  1. Counts — echo the summary counts in a brief paragraph, mention overdue urgently.
  2. MUST ACCOMPLISH — ask which tasks the user MUST accomplish today (not just want to). Wait for their answer.
  3. Confirm focus — echo the named tasks back as a numbered checklist, ask for confirmation. On confirmation, emit: →ACTION:set-focus|id1,id2,id3
  4. Close — wish a productive day.

MIT Picker Widget

When the AI emits →ACTION:set-focus, the app does not apply this directly. Instead, it renders an MITPicker message type in the chat — a checkbox list of overdue and due-today tasks. The user selects their actual MITs via checkboxes. Once tasks are selected, a Priority order panel appears below showing the selected tasks numbered in priority order with ▲/▼ buttons for reordering. The final ordered array is passed to handleMITSubmit(ids):

  1. Saves selected IDs to localStorage as gtd-todays-focus-YYYY-MM-DD.
  2. Removes the picker message from the chat.
  3. Posts the selected task list as a user chat bubble.
  4. Calls callAI again so the AI can give a final confirmation/close response.
  5. Switches the main view to the Today's Focus view.

Focus task completion is tracked: at End of Day, the app counts how many focus IDs are now marked done.

End of Day (EoD)

Phase: end → flips back to start after trigger

Pre-computed summary block

[EoD Summary]
- Focus tasks completed: 2 / 3
- Due today (not yet done): 1
- Unprocessed inbox items: 2
[/EoD Summary]

AI Conversation Flow (4 steps)

  1. Wrap-up prompt — acknowledge the day, ask: "What loose ends, new commitments, or ideas came up today that you haven't captured yet?"
  2. Capture — for each item mentioned, create an inbox task: →ACTION:create|item text|bucket:inbox
  3. Incomplete check — surface tasks due today that are not yet done, ask if any need a new due date. On confirmation: →ACTION:update|task_id|due:YYYY-MM-DD
  4. Close — brief positive close.

Context Sent to API

Full task list (all buckets). The SoD/EoD summary block is embedded in the user message, not the system prompt, so the AI sees it as data rather than instruction. No calibration context. Chat history reset to [] at the start of each daily review session.

No tools are active in daily mode — no get_task_context, no Gmail, no search.

📋 Weekly Review 6-Step GTD Review

Guides the user through the full GTD weekly review in 6 steps. Strict action-line discipline: the AI must never emit an action in the same response as a question, and at most one action per response.

Context Sent to API

Full task list (all buckets, no filter). No calibration context. Chat history carries forward across all six steps — this is a continuous multi-turn session, unlike Process mode which resets history per item.

The 6 Steps

1
Capture loose ends, new ideas, or goals

Two-phase: Phase A keep asking "Anything else?" until the user says done — no action lines during collection. Phase B after the user signals done, emit all captured items in a single response — one →ACTION:next or →ACTION:someday line per item — then move directly to Step 2 in the same response. This is the one exception to the one-action-per-response rule.

2
Process inbox to zero

Review any remaining inbox items with the user. May use →ACTION:next or →ACTION:someday to file them.

3
Review Projects

Does each project have a next action? Use →ACTION:add|title|parent:<project_id> to parent a new next action under a specific project, or →ACTION:next|title for a standalone action under UnCategorized.

4
Review Someday/Maybe

Anything ready to activate? May use →ACTION:update|id|someday:false to move to Next Actions.

5
Review Waiting For

Any follow-ups needed? Use →ACTION:waiting|title to create a new Waiting For item, →ACTION:update|id|due:YYYY-MM-DD to update due dates, or →ACTION:update|id|waitingFor:false to activate an item back to Next Actions.

6
Review Next Actions

Anything to complete or remove? May use →ACTION:update|id|done:true to mark items complete.

Action Types (Weekly Review)

Weekly review action lines are auto-applied (no confirmation bar) — the same auto-apply path as Chat, Brain Dump, and Daily Review modes.

→ACTION:next|title[|due:|defer:|effort:|notes:]

Add a standalone Next Action (goes under UnCategorized). Supports optional metadata fields.

→ACTION:add|title|parent:<project_id>

Add a Next Action as a child of a specific project. Use for Step 3 when a project lacks a next action.

→ACTION:someday|title[|due:|defer:|notes:]

Add a new Someday/Maybe item. Supports optional metadata fields.

→ACTION:waiting|title[|due:|defer:]

Create a new Waiting For item (goes under UnCategorized).

→ACTION:update|id|done:true

Mark an existing task as completed.

→ACTION:update|id|someday:true/false

Move to Someday/Maybe or activate back to Next Actions.

→ACTION:update|id|waitingFor:true/false

Move to Waiting For, or activate back to Next Actions (waitingFor:false).

→ACTION:update|id|due:YYYY-MM-DD

Set or update a due date on an existing task.

Strict Action Discipline

Critical constraints (violating these breaks the app):
  • Never emit →ACTION in the same response as a question — not ever.
  • Emit exactly one →ACTION line per response — exception: Step 1 Phase B may emit multiple →ACTION:next/→ACTION:someday lines to batch-log all captures at once.
  • Only emit when the user explicitly confirms — never if they declined or gave no clear answer.
  • If multiple changes are needed, handle them one at a time across turns — except during Step 1 Phase B batch logging.

Project Tree Structure Note

The system prompt reminds the AI that [type:category] and [type:subcategory] nodes are organisational containers — they group projects but do not require next actions themselves. Only [type:project] and [type:subproject] nodes are reviewable items. Untagged project-bucket tasks are treated as [type:project] by default. Tasks without a parent project are treated as being under the UnCategorized project.

No Auto-Advance

Note: Unlike Process Inbox, Weekly Review does not auto-advance between steps. The AI drives the step progression through the conversation. If the AI gets confused by a response, it asks a short clarifying question and continues from the same step — it does not restart or re-introduce the review.
