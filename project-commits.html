Initial commit: GTD Manager React app with AI Coach
The starting point of the entire project. A single file — src/App.jsx — containing a working GTD task manager with five buckets (Inbox, Next Actions, Projects, Waiting For, Someday/Maybe), a task list, and an AI Coach panel connected to the Anthropic Claude API. The app was initially written to run as a Claude Artifact (a self-contained React demo inside the Claude chat interface), so it had no build tooling yet.
React concept introduced — Components: Think of a React component like a shell function that returns HTML. The entire app starts as one big component called GTDManager. When you call it, it produces the full page UI. As the project grows, smaller components (TaskRow, GroupDivider, etc.) are split off — just like refactoring a monolithic script into sourced sub-scripts.

Any task can now be given a "Defer Until" date that hides it from active views until the specified date. 082a7e1 2026-04-28 fix
Fix white screen: move updateTask before advanceMetadataReview
advanceMetadataReview listed updateTask in its useCallback dependencies, but updateTask was declared ~110 lines later in the file. This caused a Temporal Dead Zone (TDZ) ReferenceError — JavaScript's equivalent of referencing a variable before it's declared — on every render, causing a white screen. The fix: move the updateTask declaration immediately before advanceMetadataReview.
JavaScript/React concept — Temporal Dead Zone: JavaScript's const declarations are not hoisted the way var is. If you reference a const before the line where it's declared (even inside a closure that captures it), you get a ReferenceError. In a React component, all const declarations with hooks run top-to-bottom on every render, so declaration order matters. This is the same as a shell script where you must define a variable before using it.

f45908c 2026-04-29 fix
fix: remove updateTask from callAI dep array (temporal dead zone crash)
Immediately after deploying the AI chat update feature, the app crashed on load with "Cannot access 'updateTask' before initialization". The cause: callAI was declared at line ~666 but updateTask was declared at line ~954 — later in the same component. Adding updateTask to callAI's useCallback dependency array caused the crash because dependency arrays are evaluated immediately when the hook is called, hitting the JavaScript temporal dead zone. The fix was to remove updateTask from the array entirely. Since updateTask uses only setTasks(prev => ...), it has an empty [] dependency array and is a stable reference that never changes — it never needed to be in callAI's deps.
JavaScript temporal dead zone (TDZ). In JavaScript, const and let declarations are "hoisted" to the top of their scope but are not initialised until the declaration is reached during execution. Trying to read such a variable before that point throws a ReferenceError. The fix here was conceptually simple — remove the reference — but the deeper lesson is: when using useCallback, check that every item in the dependency array is already initialised at the point where the hook is called.
