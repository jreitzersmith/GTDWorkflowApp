Feature Summary

A complete guide to every feature in the GTD Manager — what it does, why it was built that way, and the React/JavaScript concepts it demonstrates. Written for developers with a sysadmin background who are new to React.

About this project

GTD Manager is a personal productivity tool built as a single-page React app. It implements the full Getting Things Done (GTD) methodology — a trusted system for capturing, clarifying, organizing, reflecting on, and engaging with work. The app is powered by the Anthropic Claude API, which acts as an AI coach to help you process your inbox, review your projects, and stay on track.

Everything lives in one file: src/App.jsx (~2,900 lines). No backend. No database. State persists in localStorage. You run it with npm run dev and it opens in your browser.

🏗️ App Foundation Core

The foundation of the app: a set of shared data constants (bucket names, tag colors, effort levels), the task data model (a plain JavaScript object with about 12 fields), and the top-level GTDManager component that owns all application state and renders the sidebar navigation and main panel.

Every task is just a plain object: { id, text, bucket, done, created, priority[], location[], dueDate, effort, deferUntil, notes, childIds?, parentId? }. Tasks are stored in a tasks array in React state, which in turn is serialized to localStorage on every change.

React Concept: useState and the State Array

useState is how React remembers data between renders. Think of it like a config variable in memory — but with a special setter function. When you call the setter, React automatically re-renders the component with the new value. The app's master task list is const [tasks, setTasks] = useState([]). Every add/edit/delete goes through setTasks.

Design Decision: Everything in one component file

Rather than splitting into many files, all state lives at the top level in GTDManager and flows down to child components as props. This makes it easy to understand the full data flow at a glance — at the cost of a very long file. For a personal tool this is a reasonable trade-off.

Sysadmin Analogy

Think of useState like a variable in a shell script, except when you change it, the whole "script" re-runs from a checkpoint — only the parts that depend on that variable update in the DOM. React is basically a reactive render loop. The COLORS object is like sourcing a shared vars.sh file at the top of every script.

📥 Inbox Processing (AI Coach) AI

The Process mode of the AI coach walks you through your inbox one item at a time. For each item, it asks the AI: "What is this? What's the next action?" The AI responds with a recommended bucket (Next Actions, Projects, Waiting For, Someday/Maybe, or Trash), a brief explanation, and a suggested rewrite of the task text.

A confirmation UI appears below the AI's response with a one-click "Move to [Bucket]" button. Accepting moves the item out of Inbox and optionally creates a linked project. Rejecting keeps it in Inbox for manual handling.

React Concept: Controlled state for multi-step flows

The processing flow is a small state machine: processIndex tracks which inbox item is being reviewed. When you click "Accept", setProcessIndex(i + 1) advances to the next item. React re-renders the AI panel showing the next inbox item automatically. No loops, no manual DOM manipulation — just state changes driving the view.

React Concept: extractSuggestions() — parsing AI text

The AI returns freeform text, but the app needs structured data (a bucket name, an optional project name). The extractSuggestions() function uses regex to parse the AI's response text and pull out these fields. It's a pure function: same input always gives same output, no side effects. Pure functions are easy to test and reason about.

Sysadmin Analogy

The inbox processor is like a triage script that reads items from a queue one by one, calls an external API for a recommendation, then waits for human confirmation before moving the item to the appropriate directory. The AI is the external API. extractSuggestions() is like awk or grep -oP parsing the API response.

🤖 AI Provider Selector AI

The AI panel has a provider dropdown that lets you switch between the Anthropic Claude API (remote, requires API key) and local Ollama models (runs on your machine, no cost). The two providers use slightly different API formats: Anthropic uses its own format; Ollama speaks the OpenAI-compatible format.

The callAI() function checks the selected provider and constructs the appropriate fetch call, including the correct URL, headers, and request body shape.

React Concept: async/await with useCallback

callAI is an async function — it calls fetch(), which returns a Promise (a value that will exist later). The await keyword pauses execution until the Promise resolves. It's wrapped in useCallback so React doesn't recreate the function on every render (performance optimization). The dependency array [provider, model, apiKey] tells React to only recreate callAI when those values change.

Design Decision: Direct browser API calls

The app calls the Anthropic API directly from the browser, which requires a special header (anthropic-dangerous-direct-browser-access: true). Normally you'd proxy API calls through a backend to hide your API key. For a personal local tool, direct browser calls are acceptable — just never deploy this publicly with your key hardcoded.

Sysadmin Analogy

callAI is like a shell function that wraps curl — it knows the endpoint URL, sets the right headers, and handles both the Anthropic and OpenAI wire formats depending on a config variable. useCallback is like caching the compiled function so it doesn't get re-evaluated on every loop iteration.

🏷️ Task Metadata (Priority, Due Date, Effort) Core

Every task row shows a compact metadata panel: colored priority chips (High / Medium / Low), an optional due date, and an effort estimate. Metadata fields are editable inline via small dropdowns and date pickers that appear on demand. Tasks past their due date are highlighted in red-orange.

Priority and location are arrays (multi-select). Due date is a string in YYYY-MM-DD format. Effort is a string like "2h" or "15m" that the effortToMinutes() helper converts to numbers for sorting and summing.

React Concept: Controlled inputs

Every editable field in the task row is a controlled input — meaning its value comes from React state, not from the DOM. When you change a date picker, the onChange handler calls setTasks with the new value. React re-renders with the updated value. The DOM never holds "truth" — React state does.

React Concept: Immutable state updates (spread operator)

To update one field on one task, you can't just do task.dueDate = newDate — React won't detect that change. Instead you build a new array: tasks.map(t => t.id === id ? { ...t, dueDate: newDate } : t). The spread operator (...t) copies all existing fields; you then override just the one that changed. This is the "immutable update" pattern that React requires.

Sysadmin Analogy

Immutable updates are like creating a new config file by copying the old one and changing one line, rather than editing in place. This is intentional — React uses object identity (reference equality) to detect changes. If you mutate the existing object, React sees the same reference and skips the re-render, like a Makefile that doesn't notice a file was touched because the inode didn't change.

📍 Location Tags Core

GTD encourages tagging Next Actions with contexts — the place or tool needed to do the task (e.g., @computer, @phone, @errands). The app implements this as a multi-select "location" tag with a customizable list managed in the Location Manager settings panel. You can add your own locations, delete ones you don't use, and assign multiple locations to one task.

React Concept: Lifting state up

The list of available location options is app-level state (it needs to be shared between the Location Manager settings panel and every TaskRow). This is "lifting state up" — rather than each component managing its own copy of the data, one parent owns the array and passes it down as props. Changes in the Location Manager flow back up via a callback prop, then back down to all task rows.

Sysadmin Analogy

Lifting state up is like having a single source of truth in a config file that multiple services read. Rather than each service maintaining its own copy and risking drift, they all read from one place. The callback prop is the "write" interface — like a config management tool's API for updating that single source.

🌊 Waterfall Filtering & Group-By Core

Waterfall filtering hides Next Actions whose parent project has incomplete predecessor tasks. In GTD terms: if Task B can't start until Task A is done, hide Task B from view until Task A is completed. This keeps your Next Actions list clean — only truly available work shows up.

Group-By lets you organize Next Actions by location, project, due date, or effort. Selecting a group-by option splits the task list into labeled sections with a summary header showing the total effort for that group.

React Concept: Derived state with pure functions

waterfallFilter(tasks) and groupByField(tasks, field) are pure functions — they take the tasks array and return a transformed version without touching state. They're called directly inside the render function: const visible = waterfallFilter(tasks). There's no separate "filtered tasks" state to keep in sync. This is a key React pattern: compute derived views on the fly from the single source of truth.

Sysadmin Analogy

Pure functions for derived views are like jq filters — you pipe your JSON through them to get a view without modifying the original. You wouldn't save the jq output back to disk and maintain it separately; you just re-run the filter each time you need the view. React does the same: re-runs the filter on every render.

➕ Add Tasks to Projects Core

In the Projects view, each project row has an inline "Add task" input that lets you create a new child task directly under that project. The child task is added to the tasks array with its parentId set to the project's ID, and the project's childIds array is updated to include the new task's ID — maintaining the bidirectional parent/child link.

React Concept: Multiple state updates in one handler

Adding a child task requires updating two things at once: the new child task entry AND the parent's childIds array. In a single setTasks call, you map over the array, adding the new child object and updating the parent — all in one immutable update. React batches the re-render so the UI updates once, not twice.

Design Decision: Bidirectional parent/child links

Both the parent task (via childIds[]) and child task (via parentId) store references to each other. This redundancy makes it fast to answer both questions: "who are this project's children?" and "what project does this task belong to?" — without scanning the entire task array.

🖱️ Drag-and-Drop Reordering UI

In the Projects view, tasks can be reordered within their parent project by dragging and dropping. The drag handle (≡ icon) uses the HTML5 native drag-and-drop API. Dropping a task between two siblings reorders the childIds array of their shared parent, which immediately re-renders the tree in the new order.

The moveTaskInTree() function handles the reorder logic: find the parent, remove the dragged item from its current position, and insert it at the drop target's position.

React Concept: Recursive components (ProjectTree)

The ProjectTree component renders a task and then calls itself for each child: {task.childIds.map(id => <ProjectTree taskId={id} ... />)}. This is a recursive component. React handles the recursion naturally — each nested call is just another component instance. The tree can be arbitrarily deep without any special loop logic.

React Concept: Event handlers as props

Drag-and-drop event handlers (onDragStart, onDragOver, onDrop) are passed as props to ProjectTree. The handlers themselves live at the top level (where tasks state lives) so they can call setTasks. Child components just invoke the callbacks they receive — they never manage state themselves.

Sysadmin Analogy

A recursive component is like a directory tree traversal function that calls itself for subdirectories. The drag-and-drop reorder is like moving an entry in an ordered list file (think /etc/fstab entry order) — you remove the line from one position and splice it in at another, then save the file. moveTaskInTree() is the splice logic.

⏱️ Effort Estimates & Cumulative Totals Core

Tasks can be tagged with an effort estimate: 5m, 15m, 30m, 1h, 2h, 4h, or 8h. In the Projects view, each project row shows the cumulative effort total of all its incomplete descendants — so you can see at a glance how many hours of work a project contains. In Next Actions group-by views, each group header also shows a total effort.

The helper functions effortToMinutes() and minutesToEffortLabel() convert between the string labels and numeric minutes for arithmetic. sumDescendantEffort() recursively sums effort across the entire subtree.

React Concept: Recursive pure functions over tree data

sumDescendantEffort(taskId, tasks) is a recursive pure function. It finds the task's children, calls itself on each child, and sums the results. Because it's pure (no side effects, same output for same input), it can be called freely inside render without concern. React will call it on every render, but that's fine — it's fast and correct.

Design Decision: Effort as a human-readable string

Effort is stored as a string ("2h", "30m") rather than a number of minutes. This makes the stored data human-readable if you inspect localStorage, and it's easy to display. The conversion to minutes only happens when you need to do math — a lazy conversion pattern.

⏰ Deferred Bucket (Virtual View) New — Apr 27

Any task can have a deferUntil date. Tasks with a future defer date are hidden from their normal bucket and shown instead in the Deferred virtual view. On the morning the defer date arrives, the task "wakes up" — it's removed from Deferred and appears back in Inbox for processing.

The isDeferred(task) helper function returns true if task.deferUntil is set and is still in the future. A useEffect hook runs on app startup to auto-wake any tasks whose defer date has passed.

React Concept: Virtual views vs. stored state

The Deferred bucket is not a real bucket — there's no bucket: "deferred" value stored on tasks. Instead, tasks keep their real bucket ("inbox", "next", etc.) and the Deferred view is computed by filtering for tasks where isDeferred(task) is true. This is the "virtual view" pattern: the view is a lens on the data, not a copy of it.

React Concept: useEffect for side effects on mount

useEffect(() => { ... }, []) with an empty dependency array runs once when the component first renders (like an initialization hook). The app uses this to check localStorage on startup, load saved tasks, and immediately sweep for any tasks whose deferUntil date has passed — moving them to Inbox.

Sysadmin Analogy

The Deferred bucket is like at jobs or cron with a one-time future date. The task doesn't move into a "scheduled" queue on disk — it just has a timestamp, and a startup check (like a systemd ConditionTimestamp) decides whether to activate it. useEffect with [] is the app's equivalent of an ExecStartPre check that runs once on boot.

🔍 AI Project Review AI

The Project Review AI mode walks you through each of your projects one by one. For each project, it shows the full task list and asks the AI: "Does this project have a clear next action? Are any tasks stale or blocked?" The AI responds with concrete suggestions.

In Tasks sub-mode, the AI suggests new next actions you could add. In Metadata sub-mode, it suggests updates to existing tasks' due dates, effort estimates, and defer dates — with per-suggestion accept/reject buttons so you can review each change individually.

React Concept: Complex state for multi-step AI flows

The project review state machine tracks: which project index you're on (reviewIndex), which sub-mode you're in (reviewSubMode), the AI's current suggestions array (reviewSuggestions), and which suggestions have been accepted/rejected (reviewDecisions). Each of these is a separate useState call. When you accept a suggestion, multiple state variables update in one event handler.

React Concept: extractSuggestions and extractMetadata

Both extractSuggestions() and extractMetadata() are regex-based parsers that pull structured data out of the AI's freeform text response. extractMetadata() looks for patterns like [TASK:id][DUE:2026-05-01] in the AI's response and converts them into an array of update objects. The parser is the glue between natural language AI output and typed application state.

📊 Project Review — Metadata Mode New — Apr 28

A second sub-mode for Project Review where the AI analyzes each task's metadata and suggests concrete updates: set a due date, change an effort estimate, defer a task to a specific future date. Each suggestion appears as a card with Accept (✓) and Reject (✗) buttons. Accepting a suggestion immediately applies the change to the task in state.

React Concept: Per-item decision state

Each metadata suggestion has its own accept/reject state stored in a reviewDecisions object keyed by suggestion index. This is a common React pattern for managing granular state across a dynamic list: use an object or Map where keys are item identifiers and values are the per-item state. When any decision changes, only the affected card re-renders.

Design Decision: Structured AI output via prompt engineering

The AI is told exactly what format to use for metadata suggestions in the system prompt. This is "prompt engineering" — shaping AI output so your parser can handle it reliably. The alternative (asking the AI to return JSON) works too, but bracketed tags are more robust to the AI adding extra explanation text around the structured data.

🌳 Collapsible Project Tree New — Apr 27

In the Projects view, each project with children has a collapse/expand toggle (▶ / ▼). Three toolbar buttons allow bulk operations: ≡ Projects Only (collapse all to top-level only), ⊖ Collapse All (collapse entire tree), and ⊕ Expand All (expand everything). The collapsed/expanded state for each task is tracked in a collapsedIds Set stored in React state.

React Concept: Set as state value

A JavaScript Set (like a hashset) is used to track which task IDs are currently collapsed. This is more efficient than scanning an array. But Sets are mutable objects, so you can't just set.add(id) and call setCollapsedIds(set) — React won't detect the change (same reference). Instead, you create a new Set: setCollapsedIds(new Set([...prev, id])). Same immutability rule as arrays.

Design Decision: Collapse state is not persisted

The collapsed/expanded state resets every time you reload the page (it's not saved to localStorage). This is intentional — the last-used collapse state is ephemeral UI preference, not data. Persisting it would add complexity (serializing a Set to JSON, loading it back) for little benefit.

🔢 Descendant Count Badge UI

Each project row in the tree shows a small badge like ↓ 3 / 8 — meaning 3 incomplete tasks out of 8 total descendants. This gives you a quick at-a-glance health indicator without expanding the tree. The counts are recursive, including grandchildren and deeper.

The countDescendants(taskId, tasks) function returns an object { incomplete, total } by recursively summing across the task's children.

React Concept: Memoization with useMemo

Computing descendant counts for every project on every render could be expensive for deep trees. useMemo caches the result of an expensive computation and only recomputes when its dependencies change. Example: const counts = useMemo(() => countDescendants(id, tasks), [id, tasks]). If tasks hasn't changed, the cached value is returned immediately.

Sysadmin Analogy

useMemo is like caching the output of a slow command with $(cat /tmp/cache) and only re-running if the source file's mtime has changed. React's dependency array is the "has input changed?" check. If the check passes (no change), skip the computation and return the cached value.

🔀 Reassign Task to Different Project New — Apr 28

In the Task Detail Panel, a "Move to project" dropdown lets you re-parent a task from one project to another. This requires updating three things atomically: removing the task from the old parent's childIds, adding it to the new parent's childIds, and updating the task's own parentId.

The reassignProject(taskId, newParentId, tasks) function handles all three updates in a single tasks.map() pass, keeping the state update atomic.

React Concept: Atomic multi-record updates

When you need to update multiple tasks in one logical operation, do it in a single setTasks(tasks.map(...)) call. This guarantees the state is never in an inconsistent intermediate state (e.g., where the child has a new parentId but the old parent still lists it in childIds). React processes the update as one transaction and renders once with the fully-updated state.

Design Decision: Cycle detection with a visited Set

Before allowing a reassignment, the app checks that you're not creating a cycle (assigning a task to be a child of one of its own descendants). The check uses a visited Set to track nodes already seen during tree traversal — the standard way to detect cycles in a graph. Without this, the recursive tree renderer would loop infinitely.

✏️ Inline Title Editing UI

Each task title has a pencil (✏️) icon that toggles the title into an editable text input. Pressing Enter or blurring the input saves the change. Pressing Escape cancels without saving. The edit state is local to the task row component — it doesn't propagate to the parent.

React Concept: Local component state vs. app state

Whether a title is being edited is local state — only the TaskRow needs to know. It's stored with a local useState inside TaskRow, not in the top-level GTDManager. The edit buffer (the in-progress text) is also local state. Only when the user confirms the edit does the change propagate up to app state via a callback. This keeps unrelated state isolated and the UI responsive.

React Concept: useRef for focusing inputs

When the pencil icon is clicked, the title input should immediately receive keyboard focus. useRef gives you a direct reference to a DOM element so you can call .focus() on it imperatively. This is one of the few cases where "reaching into the DOM" is the right approach — focus management is inherently imperative and doesn't fit the declarative state model.

🏷️ Tag Display Toggle UI

A settings toggle controls how tags (priority, location) are displayed in task rows: either as colored chips (colored pill badges) or as compact dot indicators. The dot view saves horizontal space; the chip view is more readable. The preference is saved to localStorage.

The task row renders either the chip view or the dot view based on a boolean setting: {tagDisplay === 'chip' ? <ChipView /> : <DotView />}. 📋 Task Detail Panel New — Apr 28

Clicking any task title opens a 360px side panel on the right side of the screen. The panel shows the full task: an editable title, a multi-line notes textarea (autosaves on blur), all metadata fields (due date, defer date, effort, locations), a bucket move dropdown, and complete/delete buttons. The panel closes via ×, Escape, or when you change the bucket.

Only one panel can be open at a time, controlled by a single selectedTaskId state value. The panel is rendered conditionally: {selectedTaskId && <TaskDetailPanel />}.

React Concept: Conditional rendering with &&

The short-circuit && pattern is the idiomatic way to conditionally render something in React: {selectedTaskId && <TaskDetailPanel taskId={selectedTaskId} />}. If selectedTaskId is null, the expression short-circuits and nothing renders. If it's set, TaskDetailPanel renders. No if-statement needed.

React Concept: Autosave with onBlur

The notes textarea in the detail panel uses onBlur (fires when the element loses focus) rather than onChange (fires on every keystroke) to save notes. This avoids a setTasks call on every character typed — saving only when the user tabs away or clicks elsewhere. The in-progress text is held in local component state as a buffer.

Sysadmin Analogy

The detail panel is like a modal config editor — it reads the current config (task state), lets you edit all fields, and writes back only when you're done (onBlur autosave). selectedTaskId is like a file descriptor or lock — there's only one open at a time, and closing it (setting to null) releases it.

↔️ Resizable Panels New — Apr 29

The three main layout regions — sidebar, AI coach panel, and task detail panel — are all resizable by dragging a handle between them. Dragging the handle fires mouse events that update a width/height value in state, which immediately re-renders the layout with the new dimensions. Panel sizes persist to localStorage so they survive page reloads.

A custom hook useResizer(defaultSize, storageKey, min, max) encapsulates all the drag logic: attaching/removing mouse event listeners, clamping the value to min/max bounds, and persisting to localStorage.

React Concept: Custom hooks

A custom hook is a regular JavaScript function whose name starts with use and which calls one or more React hooks internally. useResizer calls useState (to hold the current size) and useEffect (to manage the global mousemove/mouseup event listeners). You call it like: const [sidebarWidth, sidebarHandlers] = useResizer(240, 'sidebarWidth', 160, 400). Custom hooks let you extract and reuse stateful logic without copy-pasting.

React Concept: useEffect cleanup functions

When you add a global mousemove listener in a useEffect, you must remove it when the component unmounts (or when the effect re-runs). The cleanup function is the return value of the effect: useEffect(() => { window.addEventListener('mousemove', handler); return () => window.removeEventListener('mousemove', handler); }, [...]). Without cleanup, you'd leak event listeners every time the component re-renders.

Sysadmin Analogy

A custom hook is like a reusable shell function or a Bash library you source into multiple scripts. It bundles related state and behavior so you don't duplicate it. The useEffect cleanup is like a trap on EXIT — it ensures you clean up resources (event listeners) when you're done, just like a trap removes temp files when a script exits.

📧 AI Coach — Gmail Integration (OAuth2 PKCE) New

The AI Coach can now read the user's Gmail inbox. When Gmail is connected and the user asks a question in Chat mode, Claude can call the gmail_search tool to query Gmail and retrieve message metadata (From, Date, Subject, Snippet) for up to 5 matching messages. The tool is only injected into the tool-use loop when a valid Google access token is present — it's invisible to users who haven't connected Gmail.

Authentication uses OAuth2 PKCE (Proof Key for Code Exchange), which allows a browser-only app to obtain a Google access token without ever exposing a client secret in the network request. A random 64-byte code verifier is generated via crypto.getRandomValues, hashed with SHA-256 to produce a code challenge, and sent with the authorization request. Google validates the verifier against the hash during the token exchange. A random state parameter prefixed with gtd_ provides CSRF protection and discriminates Google callbacks from Supabase magic-link codes.

A Connect Gmail / Disconnect button lives in the Settings panel. When not connected, any OAuth error (wrong client, expired PKCE, token exchange failure) is displayed inline below the connect button so the user can diagnose the problem without opening DevTools.

Key implementation details

Several tricky problems were solved to make this work reliably:

Supabase ?code= collision. Supabase v2's createClient runs at module load time — before React renders — and scans window.location.search for a ?code= parameter to attempt a PKCE session exchange. This would silently consume our Google OAuth code. Fix: pass { auth: { detectSessionInUrl: false } } to createClient, then manually call supabase.auth.exchangeCodeForSession(code) in the auth useEffect for codes that don't have a gtd_ state prefix (those are Supabase magic-link codes).

React StrictMode purity. React 18's StrictMode double-invokes useState initializer functions to detect side effects. The first call read and removed the PKCE entry from localStorage; the second call found it gone and returned null — React used the second result as the final state value, so pendingGoogleAuth was always null. Fix: the useState initializer is now pure (read-only, no removeItem). The useEffect that performs the token exchange uses an atomic claim pattern — it calls localStorage.getItem as a mutex: the first StrictMode run claims and removes the entry; the second run finds it gone and exits immediately, preventing a duplicate exchange request.

sessionStorage vs. localStorage. PKCE data was originally stored in sessionStorage, which browsers can clear across cross-origin navigations (the Google auth redirect). Switched to localStorage with a 5-minute expiry timestamp.

Desktop app credential + client_secret. Google requires a client_secret in the token exchange request even for Desktop app credentials. The app uses the Desktop app credential type (which does not require registering redirect URIs for localhost) and passes VITE_GOOGLE_DESKTOPCLIENT_SECRET in the token exchange body.
